# 素页 (Suyep) MVP 产品设计文档

> 域名：suyep.com（已持有）
>
> 寓意：素 = 留白、素雅、不加修饰；页 = 一城一页，护照集邮的视觉隐喻。
>
> 适配 P0 用户（25-32 岁女性，小红书重度，手账癖）的审美调性。

## 一句话定义

一个"有你记忆的城市地图"——点亮去过的城市，每个城市有你的照片、AI 专属印章和到访日期。

## 核心痛点

高德地图的城市点亮功能只展示城市的公共信息，没有任何与用户个人相关的记忆。用户点亮了一个城市，点开后却看不到自己的照片、自己的故事。"素页"要解决的是：让每一个点亮的城市都承载属于你的记忆。

## 目标用户（STP 分层）

### P0 核心种子用户（MVP 0-2 月，<1000 人）

**B × D 交集**：小红书重度用户 ∩ 手账/集邮癖人群 ∩ 中频旅行者（年 4-8 次）。

具象画像：
- 25-32 岁女性为主（70%+ 占比）
- 一二线城市（北上广深杭 + 成都西安）
- 互联网 / 广告 / 媒体 / 咨询 / 教培 / 设计行业
- 年薪 15-40 万，未婚或已婚未育
- 年出行 4-8 次：2 次小长假长途 + 3-5 次周末周边
- 旅行后必整理照片、发小红书笔记 / 朋友圈九宫格
- 收藏倾向：景区爱盖印章、囤文创明信片 / 冰箱贴、试过纸质旅行手账（多半坚持不下来）
- 关注"仪式感"、"治愈"、"复古"、"国潮"、"少女心"等小红书标签

核心心理需求（产品要击中的）：
1. 手机几千张照片找不到 → "一城一图"满足整理欲
2. 朋友圈 / 小红书发啥都同质化 → 印章海报够独特
3. 想做手账但坚持不下来 → AI 生成降低门槛
4. 旅行"过了就过了"没沉淀感 → 集齐印章 = 有积累

### P1 增长引擎（v0.3 后）

**B 类全集**：中频小红书分享型旅行者（约 1.5-2.1 亿）。靠 P0 种子用户在小红书产生的 UGC 自然外扩，不投放，不主动获客。

### P2 节点 campaign 触点（不开发功能）

**C 类情侣 / 闺蜜**：仅在情人节 / 520 / 七夕等节点上线"双人版海报文案"——数据底层是单人产品，文案改双人表达。**不开发双人功能**，双人功能延迟到 v3.0。

### 自然覆盖（不主动获客）

**A 类高频旅行者**：包括用户本人，作为 dogfood。产品逻辑天然服务这群人，但获客成本高（已被高德 / 世界迷雾占据），不主动投放。

### 明确放弃

**E 类出差族**：心理场景与产品调性不符（"出差点亮"无社交价值），不做。

### 选择 B × D 而非其他组合的理由

- 报告 1 推 "B + C 双轨"的问题：C 类核心价值在双人共同记忆，但双人功能在 v3.0，MVP 期功能层接不住，会拉低留存
- 报告 2 推 "B × D 交集"的优势：B 提供规模化外扩潜力，D 提供 MVP 阶段最珍贵的天使用户特质（容忍粗糙、提供高质量反馈、自然产生 UGC、付费意愿天花板高）
- 4.6 把 C 提到 P0.5 是混淆"用户分层"与"营销节奏"——P0.5 应理解为节日 campaign 同步推进，不是功能优先级

## MVP 范围

### 做什么

- 中国 / 世界地图，去过的城市高亮显示
- 上传照片自动识别城市（读取 EXIF GPS + 逆地理编码）
- 每个城市生成一枚 AI 艺术印章（4 种风格：国潮 / 手账 / 治愈 / 复古）
- 城市记忆卡片：一张代表照片 + AI 印章 + 到访日期
- 护照集邮页面（翻页浏览所有印章）
- 旅行统计数据（总城市数、覆盖省份、累计公里、最远到达等）
- 生成分享海报（双版本：9:16 朋友圈 + 3:4 小红书）
- 旅行中支持手动打卡（获取当前定位 + 拍照）
- 手机号验证码登录
- 小程序订阅消息（48h 内提醒用户回来整理旅行）
- 邀请码灰度发布机制

### 不做（MVP 阶段）

- 多人 / 双人 / 家庭功能（v3.0）
- 完整旅行故事 HTML 页面（v1.0 简版，v2.0 完整）
- AI 攻略生成
- 实体打印下单（Phase 2）
- 用户社区 / 评论 / 关注
- 任何付费功能（MVP 完全免费）

### 平台

**Taro 双端**：同一份代码同时编译为 Web H5 + 微信小程序，共享 Supabase 后端。

- **H5 端**：主开发路径，EXIF 解析直接在浏览器跑（exifr 库一行解决）
- **小程序端**：微信内传播主路径，分享卡片体验好；EXIF 解析走"上传后云端处理"兜底

为什么不只做小程序：
- 小程序 `wx.chooseImage` 默认不返回 EXIF 信息，需自行用 Canvas 解析二进制或上传后云端处理，远不如浏览器 exifr 直接
- 小红书禁外链跳转，截图二维码扫码可走两端任一
- 个人主体小程序不能开通微信支付（硬限制），双端架构让 H5 在过渡期不被卡住

## 用户流程

### 流程 A：旅行后补录（主流程）

1. 用户点击"上传照片"
2. 选择多张旅行照片
3. 客户端读取每张照片的 EXIF（GPS 坐标 + 拍摄日期）
4. 调用高德逆地理编码 API（结果全局缓存），坐标转为城市名
5. 按城市分组展示："检测到你去过 厦门(5.10-5.12, 47张)、泉州(5.13, 12张)"
6. 用户为每个城市选一张代表照片
7. 用户选择印章风格（4 种）
8. 系统返回印章：缓存命中即时返回；未命中走异步队列 + 占位印章
9. 城市在地图上点亮，记忆卡片生成完毕

### 流程 B：旅行中现场打卡

1. 用户点击"现场打卡"
2. 系统获取当前 GPS 定位，识别城市
3. 用户拍照或从相册选一张
4. 选择印章风格 → 印章生成（同上，缓存优先）
5. 城市点亮

### 流程 C：分享

1. 用户在地图上点击已点亮城市
2. 查看城市记忆卡片（照片 + 印章 + 日期）
3. 点击"分享" → 选择海报版本（朋友圈 9:16 / 小红书 3:4）
4. 海报包含：地图缩略图 + 照片 + 印章 + 日期 + "我的第 X 个城市"+ 回流二维码
5. 长按保存 / 微信原生分享 / 截图发小红书

### 流程 D：48h 触发（防止旅行后拖延整理）

1. 用户上传第一张带 GPS 的照片时，弹出"开启提醒，回家后帮你整理记忆"
2. 用户授权小程序订阅消息（每授权一次只能推送一次）
3. 拍照后 48h 内用户未上传，系统通过订阅消息提醒
4. 提醒文案："你昨天的厦门之行还没整理哦～现在打开「轨迹」一键收集印章"

## 页面结构

### 整体导航：底部三个 Tab

```
📍 地图    |    📖 护照    |    🎞️ 记忆
```

- **地图 Tab**：城市点亮地图，打卡入口
- **护照 Tab**：印章集邮页面（按城市，轻量收集感）
- **记忆 Tab**：旅程故事列表（按旅程，丰富沉浸感）

### 核心概念区分

| | 印章（护照 Tab） | 记忆（记忆 Tab） |
|---|---|---|
| 粒度 | 每个**城市**一个 | 每次**旅程**一个 |
| 形态 | 一张印章图片 | 完整 HTML 页面（多照片+文案+路线） |
| 感觉 | 轻量、快速、收集感 | 沉浸、丰富、回味感 |
| 类比 | 护照上的签证章 | Apple 照片的"回忆" |

### 页面 1：地图 Tab（首页）

- 全屏地图（高德 JS API / 小程序 SDK）
- 去过的城市用暖色调圆点标记（避免冷色调，匹配少女心调性）
- 右上角统计数字："已点亮 X 个城市"
- 底部两个按钮："上传照片"（旅行后）/ "现场打卡"（旅行中）
- 点击已点亮城市 → 弹出城市卡片（代表照片+印章+日期）

### 页面 2：护照 Tab（印章集邮）

- 翻页效果（像真实护照一样左右滑动）
- 每页 2-4 个印章，按时间顺序排列
- 顶部统计："已收集 X/337 城"（337 = 中国地级市数量）
- 长按某个印章 → 查看该城市的记忆卡片
- 可切换视图：护照翻页模式 / 网格九宫格模式

### 页面 3：记忆 Tab（旅程故事列表）

- 按时间倒序展示所有旅程
- 每条旅程卡片：封面照片 + 旅程名称 + 日期 + 途经城市标签
- 点击进入旅程详情（HTML 故事页面）
- MVP 阶段：此 Tab 显示"即将上线"占位
- v1.0 阶段：简版旅程页面（照片时间轴 + 地图路线）
- v2.0 阶段：完整版（AI 文案 + 背景音乐 + 翻页动画）

### 页面 4：上传识别页

- 照片选择区域
- 识别结果分组展示（按城市，自动归为一次旅程）
- 每个城市可选代表照片
- 印章风格选择（4 种风格的预览缩略图）
- "生成印章"按钮 + 加载动画

### 页面 5：城市记忆卡片（弹窗 / 抽屉）

- 代表照片（大图，圆角）
- AI 印章（居中展示）
- 城市名 + 到访日期
- "分享"按钮（弹出双版本海报选择）
- "查看旅程"按钮（跳转到该城市所属旅程的记忆页面）

### 页面 6：分享海报生成页

- 海报预览（用户先选比例：朋友圈 9:16 / 小红书 3:4）
- "保存到相册"按钮
- "分享给朋友"按钮（小程序端走原生分享卡片）
- 海报底部嵌入小程序码（支持小红书截图传播 → 他人扫码回流）

### 页面 7：个人设置

- 头像 + 昵称
- 设置：修改昵称、退出登录、订阅消息授权管理
- "愿付意愿问卷"入口（v0.3 加，假按钮模式）

### 统计数据系统

在护照 Tab 顶部 + 分享海报中展示：

- 已点亮城市数 / 337（中国地级市总数）
- 覆盖省份数 / 34
- 累计旅行公里数（基于城市间直线距离估算）
- 最远到达的城市（距离家乡最远的打卡点）
- 最北 / 最南 / 最东 / 最西打卡点
- 今年新增城市数
- 连续打卡月数

统计数据同时用于分享海报，增强"炫耀价值"——如"我已走过中国 15% 的城市，最远到达拉萨（距家乡 3,200km）"。

## 设计语言（小红书化）

P0 用户 70% 女性，调性必须匹配。所有视觉决策遵循以下原则：

### 配色

- 主色：暖色调（莫兰迪、奶油色、低饱和粉 / 绿 / 蓝）
- 禁用：纯黑背景、霓虹色、赛博朋克色谱
- 地图皮肤：暖灰调而非冷蓝调

### 排版

- 大量留白
- 圆角（卡片 16px 圆角，按钮 8-12px）
- 字体：思源黑体（中文）+ Inter（西文），免费商用
- 字重：偏细（Regular / Light），少用 Bold

### 文案语气

- 第一人称（"我的"轨迹，不是"用户的"轨迹）
- 温柔、亲切、不油腻
- 例："你又点亮了一座城" 而非 "成功添加新足迹"
- 例："这枚印章已经为你保留" 而非 "印章已生成"

### 印章风格（4 种）

| 风格 | enum 值 | 关键词 | 适合城市 / 场景 |
|---|---|---|---|
| **国潮** | guochao | 传统纹样、朱砂色、书法字体 | 西安、北京、苏州 |
| **手账** | shouzhang | 水彩、贴纸感、手绘线条 | 厦门、大理、成都 |
| **治愈** | zhiyu | 莫兰迪、几何、留白 | 杭州、青岛、东京 |
| **复古** | fugu | 烫金边框、护照章质感、深棕红 | 巴黎、伦敦、上海 |

**砍掉**：原 spec 的"赛博朋克"风格——与目标人群女性向调性不匹配。

### 海报双版本

| 版本 | 比例 | 用途 |
|---|---|---|
| 朋友圈版 | 9:16 | 微信朋友圈、抖音壁纸 |
| 小红书版 | 3:4 | 小红书笔记封面（小红书原生比例）|

两版本共用同一套素材（地图缩略图 + 照片 + 印章 + 日期 + 统计数据），仅排版差异。

## 首发城市清单（Top 20）

MVP 上线时这 20 城市预先设计高质量印章模板（4 种风格 × 20 城 = 80 张），存 CDN，AI 调用 0 次。

**国内 17 城**：厦门、成都、长沙、西安、大理、青岛、苏州、杭州、北京、上海、丽江、重庆、广州、深圳、三亚、南京、武汉

**国内补充 3 城**：哈尔滨、敦煌、香港

**选择标准**：小红书旅行笔记数 Top 20 城市，覆盖 P0 用户高频去的目的地。其他城市走 AI 实时生成 + (city, style) 全局缓存。

## 技术架构

### 技术栈

| 层级 | 选择 | 理由 |
|------|------|------|
| 前端框架 | **Taro 3.x（React 模式）** | 一码两端：H5 + 微信小程序 |
| UI 库 | Tailwind CSS（H5）+ NutUI-Taro（小程序）| 双端样式适配 |
| 后端 / 数据库 | Supabase | 一站式：PostgreSQL + Auth + Storage + Edge Functions |
| 地图 | 高德地图 JS API v2（H5）/ 小程序 SDK | 中国地图体验好 |
| EXIF 解析 | exifr（H5 主路径）/ 上传后云端处理（小程序兜底）| 小程序 wx.chooseImage 不返回 EXIF |
| AI 印章生成 | 通义万相 API（主）/ Replicate FLUX（备）| 国内稳定，中文城市理解好 |
| 图片压缩 | browser-image-compression（H5）/ 微信原生 API（小程序）| 上传前压缩到 1MB |
| 海报生成 | html2canvas（H5）/ canvas 2d 小程序原生 | 前端将 DOM 渲染为图片 |
| CDN | 阿里云 CDN | 静态资源 + 预生成印章 |
| 部署 | Vercel（H5，绑定 suyep.com）+ 微信小程序后台 | H5 免费，小程序需个体工商户主体 |

### 分层防爆架构

```
用户请求
   ↓
┌─────────────────────────────────────┐
│ [CDN 层] 阿里云 CDN                  │ 80%+ 流量在此消化
│ - 静态资源（HTML / CSS / JS）         │
│ - Top 20 城市预生成印章（80 张）       │
│ - 长尾印章缓存                        │
└─────────────────────────────────────┘
   ↓ cache miss
┌─────────────────────────────────────┐
│ [客户端] H5 + 小程序                 │ 重计算尽量在客户端
│ - EXIF 解析、图片压缩、海报渲染        │
│ - 邀请码校验前置                      │
└─────────────────────────────────────┘
   ↓ API 请求
┌─────────────────────────────────────┐
│ [API 层] Supabase Edge Functions    │ 限流：5 req / 分 / 设备
│ - 邀请码闸门                         │
│ - 高德 API 全局缓存（GPS → 城市）      │
│ - 印章查询：先查 StampCache 再决定生成 │
└─────────────────────────────────────┘
   ↓
┌─────────────────────────────────────┐
│ [数据层] Supabase PostgreSQL         │
│ - 城市足迹、照片、用户                │
│ - StampCache（全局共享，所有用户复用） │
└─────────────────────────────────────┘
   ↓ cache miss only
┌─────────────────────────────────────┐
│ [AI 队列] 异步任务                    │
│ - 通义万相 API（主）/ FLUX（备）       │
│ - 失败 → 兜底模板印章                 │
│ - 完成后写回 StampCache 供全局复用     │
└─────────────────────────────────────┘
```

### 防爆三件套

**1. 印章 (city, style) 全局缓存共享**

同一城市同一风格的印章，全平台所有用户共享一张图。1000 个用户去同一个小县城，AI 只生成 1 次。

代价：失去"每人独有印章"卖点。换得：AI 调用量降低 100x+，成本和稳定性双赢。
折中：基础印章共享，海报上叠加用户的访问日期 / 编号体现个人化。

**2. 邀请码灰度发布**

- 前 1000 名内测无门槛
- 第 1001 个起开始要邀请码
- 每个老用户可发 3 个邀请码
- 同时制造 FOMO 传播效应（西瓜足迹反例的相反操作）

**3. 服务降级开关**

一个全局 feature flag：紧急时关闭 AI 生成，所有印章统一走兜底模板。爆发期一键切换，保住基础体验。

### AI 印章生成策略

- **Top 20 热门城市**：4 风格 × 20 城 = 80 张预生成印章，CDN 永久缓存
- **其他城市首次访问**：异步生成 + 占位印章（基础模板）+ 完成后写入 StampCache
- **生成失败**：自动 fallback 到通用兜底模板（城市名 + 日期的简约印章），保证体验不中断
- **prompt 模板**：`"一枚圆形旅行印章，{city_name} 主题，{landmarks} 地标元素，{style} 风格，白色背景，适合收藏，不含真人脸、品牌 logo、版权角色"`

### 监控与告警

| 指标 | 阈值 | 动作 |
|---|---|---|
| 每日 PV | 超 10 万 | 短信告警 |
| AI 调用量 | 单小时超 1000 次 | 短信告警 + 推荐降级 |
| 错误率 | 5xx 超 1% | 短信告警 |
| Supabase 存储 | 超 80% 容量 | 触发清理 / 升级提示 |

### 印章设计原则

- **绝对禁用**：真实建筑摄影、真人头像、品牌 logo、政治符号、宗教符号
- **可用元素**：城市文化符号（兵马俑 → 陶俑剪影艺术化，外滩 → 线条勾勒）、自然景观（海浪、山形）、抽象图案
- **风格统一**：4 风格之间需保持设计语言连贯，不能国潮风看像奶茶店、治愈风看像殡仪馆

## 关键设计决策

### 1. 冷启动引导

首次打开时引导用户："选择你的家乡，点亮第一个城市"。家乡不需要照片验证，确保用户一进来地图就不是空的。

### 2. 无 GPS 照片处理

上传照片时如果检测不到 GPS 信息：
- 弹出提示："这张照片没有定位信息"
- 提供手动选择城市的选项
- 不默默丢弃，给用户补充信息的机会

### 3. 同一城市多次到访

同一城市只有一条足迹记录。多次到访时：
- 更新 `latest_visit_date`
- 新照片追加到该城市的照片集合
- 用户可以随时更换代表照片
- 印章保留首次生成的（除非用户主动重新生成）

### 4. 照片压缩策略

- 客户端上传前自动压缩至 1MB 以内（保留 EXIF）
- 生成缩略图（200x200）用于列表展示
- 原图用于卡片大图和海报

### 5. 登录方式

- MVP 使用手机号 + 短信验证码（Supabase Auth 原生支持）
- 小程序端额外接入微信 OAuth（用户授权后自动获取 unionid，与手机号关联）
- 两端通过手机号关联同一账户

### 6. 48h 触发机制

- 用户上传第一张带 GPS 的照片时，引导授权小程序订阅消息
- 拍照后 48h 内未上传，触发一次提醒推送
- 这是 MVP 期产品最重要的留存抓手（旅行后拖延整理是核心流失点）
- 公众号模板消息这条路不走（订阅号不支持，且个体工商户订阅号开不了赞赏）
